## 1) Executive Summary

**Goal:** Build an interactive, AI‑powered economic dashboard that (a) shows historical trends, (b) produces short‑term forecasts, (c) enables “what‑if” scenario analysis, (d) explains forecasts in **plain language with citations**, and (e) issues **alerts on significant deviations**.

**What’s novel:**  
- **Multi‑Lens Explainability Cards** generated by a Bedrock LLM, **constrained by retrieved evidence** (RAG) from **FRED/BEA** releases and metadata. The cards cover: **Drivers, Evidence, Counterfactual, Confidence**, and include inline citations and snippets. Grounding is implemented with **Bedrock Knowledge Bases** + **OpenSearch Serverless vector store** (no external vendors).   
- **Revision‑aware forecasting** (optionally using ALFRED vintage data) and **counterfactual simulation** on exogenous variables (e.g., unemployment, CPI).   
- **Explainability at time‑series granularity** using **SageMaker Clarify’s time‑series Shapley attribution**, feeding the **Drivers** lens.

```
│ Clarify for feature attribution (SHAP-like)  │
│ RCF anomaly scores for deviations            │
└──────────────────────────────────────────────┘
           ↓
┌──────── Explainability & RAG Layer ──────────┐
│ Bedrock Knowledge Base (Titan Embeddings)    │
│ Vector store: OpenSearch Serverless          │
│ LLM (Claude via Bedrock) w/ citations        │
│ Prompt orchestration & guardrails            │
└──────────────────────────────────────────────┘
           ↓
┌────────────── API & Orchestration ───────────┐
│ EventBridge (schedules & alerts), SNS/Email  │
│ Lambda/Step Functions for jobs               │
│ REST/GraphQL APIs for UI                     │
└──────────────────────────────────────────────┘
           ↓

│ Trend Explorer | Forecasts | Scenario Studio │
│ Multi‑Lens Explainability Cards              │
│ Alert Console & Subscriptions                │
└──────────────────────────────────────────────┘
```

**Bedrock Knowledge Bases** provide retrieval‑augmented generation (RAG) with **citations**, and can index multi‑modal documents; **OpenSearch Serverless** is the AWS‑recommended vector store for Bedrock RAG scenarios.   
**Anthropic Claude** models are natively available in Bedrock and support large contexts and structured prompting; we’ll use Claude Sonnet for cost–performance balance.

## 3) Data & Evidence Layer

### Sources & Access
- **FRED/ALFRED:** Pull historical observations and **vintage** (revision) histories per series; use release endpoints for calendars.   
- **BEA:** Use BEA API for NIPA tables and GDP detail; register a free API key; leverage dataset/parameter metadata to discover series. 

### Ingestion & Storage
- **Lambdas on EventBridge schedules** call FRED/BEA APIs; raw JSON saved to **S3**, converted to **Parquet**; **Glue Catalog** tables enable SQL via Athena.   
- **Release Calendar Table:** Normalize FRED `releases/dates` and BEA RSS to pre‑populate upcoming release events and annotate series with next expected update (drives alerts and data freshness banners). 

### Metadata & Provenance
- Maintain **series metadata** (frequency, seasonal adjustment, units, source link, BEA table/SeriesCode) to preserve **provenance** for Evidence citations.

## 4) Forecasting Layer (Short‑Term Forecasts)

### Option A — Amazon Forecast (if account already has access)
- Use **AutoPredictor** (per‑series best algorithm ensemble) for probabilistic forecasts (p10/p50/p90), plus **Explainability reports** (impact scores). Inputs: target series, **related time series** (exogenous), **item metadata**.   
- Note: **Amazon Forecast is no longer available to new customers**; retain this path if your AWS account is eligible, otherwise use Option B. 

### Option B — SageMaker native pipeline (recommended robust path)
- **Models:**  
  - **DeepAR / Prophet / ARIMA / ETS** via SageMaker training or integrated frameworks (e.g., GluonTS) to cover seasonal, autoregressive, and neural baselines. (Algorithm references mirror Forecast’s families.)   
  - **Chronos/ChronosX research** informs incorporation of **covariates** (exogenous variables) for scenario analysis; we adapt the idea (past/future covariate blocks) in our pipeline design. 
- **Explainability:** Use **SageMaker Clarify** time‑series **asymmetric Shapley** methods to compute per‑feature, per‑lag contributions (**Drivers** lens). 
- **Deviations:** Run **Random Cut Forest (RCF)** inference on residuals or observed series to flag **anomalies** (spikes/structural breaks) for alerting and to annotate Confidence. 
- **Uncertainty:** Output **quantile forecasts** (p10/p50/p90) and ensemble variance to drive **Confidence** cards, consistent with probabilistic forecasting best practices. 

**Data for exogenous variables (“related time series”):** unemployment, CPI, PCE deflator, policy rates, etc. We treat these as **forward‑looking** when schedules provide near‑term projections (where available) and otherwise as historical drivers. (The schema distinction mirrors Forecast’s **historical vs forward‑looking** related series.)

## 5) Scenario Studio (What‑If / Counterfactuals)

**Purpose:** Let analysts perturb exogenous inputs and see the forecast update with quantified impact.

**Design:**
- **Scenario bundle:** A JSON object mapping indicator → shock profile (e.g., “CPI +0.6% annualized next 2 quarters”, “Fed Funds −50 bps next meeting”).  
- **Simulation engine:**  
  - Re‑forecast with **modified covariate trajectories** (future covariates treated as “forward‑looking”).   
  - Compute **local sensitivities** from Clarify attributions and finite‑difference runs to populate **Counterfactual** lens (e.g., “If unemployment were 0.3pp higher, GDP p50 in Q3 would drop ~0.2%”). 
- **Revision awareness:** Optionally run scenarios at a chosen **vintage date** (ALFRED) to reflect “what was known then.”

## 6) Multi‑Lens Explainability Cards (LLM‑generated, Evidence‑grounded)

Each forecast refresh produces a **card** with 4 lenses:

### A. **Drivers**
- Inputs: Clarify **feature/lag attributions** for the forecasted horizon; model‑level contributions aggregated per indicator (e.g., CPI, unemployment, rates).  
- Output: A ranked list (“Top drivers”), quick charts, and **lay‑friendly summary** synthesized by Claude. 

### B. **Evidence**
- RAG retrieves **snippets** from **BEA releases**, **FRED series notes**, and **release calendars**; the LLM composes the explanation with **inline citations** to source URLs and series IDs. **Bedrock Knowledge Bases** automatically include **citations** and can ingest documents with images, enabling the card to embed release tables/figures. 

### C. **Counterfactual**
- Using scenario sensitivities: “Minimal changes” that would **materially shift** the p50 forecast (e.g., “Core CPI +0.2pp next month flips inflation p90 above 3.5%”). Content curated by LLM but **numerically grounded** in recomputed runs. 

### D. **Confidence**
- Shows **uncertainty bands** (p10/p50/p90), **ensemble agreement** (model consensus score), and **anomaly flags** (RCF) with thresholds. The LLM provides a plain‑language confidence summary (“Forecast is robust; models agree; no structural anomalies detected”). 

**Guardrails & Prompting:**  
- Use **Claude on Bedrock** with system prompts that enforce: “Respond only with retrieved evidence snippets; include citations; avoid conjecture.” Claude supports large context and structured outputs; we keep responses short and evidence‑rich.

## 7) RAG & Knowledge Base Design (Evidence Grounding)

- **Document corpus:**  
  - **FRED** series metadata pages (cached), **release notes**, **series descriptions**;  
  - **BEA** tables documentation, release PDFs/RSS, dataset parameter docs.   
- **Embedding & store:** **Titan Embeddings**; **OpenSearch Serverless** vector index for similarity search; **metadata filters** by indicator, frequency, release date.   
- **Retrieve‑and‑generate pipeline:** Query → top‑k chunks + rerank → pass as context to Claude → **citations** returned. Bedrock Knowledge Bases provide managed ingestion, chunking, and **citation support** out of the box. 

---

## 8) Alerts for Significant Deviations

- **Event sources:**  
  - **Forecast deviation:** Actual vs p50 crossing a configured threshold;  
  - **Anomaly:** RCF score > Nσ;  
  - **Data change:** New release ingested for watched indicators. 
- **Routing:** **EventBridge rules** evaluate deviations and trigger Lambda/SNS notifications (email/Slack webhook); **Scheduler** can also send **digest summaries** daily/weekly.   
- **Noise control:** Use consolidation Lambda to group related alerts and suppress flapping (pattern aligns with event‑driven alerting best practices).

## 9) Dashboard UX (Pages & Interactions)

1. **Trend Explorer**  
   - Select indicator (CPI, GDP, unemployment, etc.) → historical line/seasonality decomposition → **data freshness** & **next release** badge. (Sources: FRED/BEA, release calendars.) 

2. **Forecasts**  
   - Horizon selector (3–12 months/quarters), **p10/p50/p90 bands**, **model consensus score**, **residuals chart**, **anomaly markers**. 

3. **Scenario Studio**  
   - Compose shocks on exogenous variables; run simulations; compare baseline vs scenario; export scenario bundle.

4. **Explainability Cards**  
   - Four lenses with collapsible **Evidence snippets** (links to FRED/BEA pages), mini‑charts, and an “**Why the model thinks this**” plain‑language paragraph; every claim cites sources. 

5. **Alerts & Subscriptions**  
   - Subscribe per indicator; view recent deviations; acknowledge and annotate.

## 10) Data Schemas (S3/Glue)

- **`timeseries_curated`**:  
  - `indicator_id`, `source` (FRED/BEA), `vintage_date` (optional), `timestamp`, `value`, `unit`, `seasonal_adjustment`, `release_id` (FRED), `bea_table` (BEA). 
- **`exogenous_curated`**:  
  - same keys + `is_forward_looking` flag for scenario editor (mirrors Forecast’s related time series concept). 
- **`release_calendar`**:  
  - `source`, `release_name`, `next_release_date`, `series_ids[]`. 
- **`explainability_attribs`**:  
  - `indicator_id`, `horizon_t`, `feature`, `lag`, `attribution_score` (Clarify outputs). 
- **`anomaly_scores`**:  
  - `indicator_id`, `timestamp`, `rcf_score`, `threshold_exceeded`.

## 11) Generative AI (Bedrock) Prompting Strategy

- **System prompt (Claude):**  
  “You are an economic forecaster assistant. Use **only** provided context. For each claim, include a citation (URL + series/table id). Prefer concise, plain language. If evidence is insufficient, say so.”

- **Context assembly:**  
  - Top‑k **RAG chunks** (BEA/FRED releases, series notes) + **numerical summary** (drivers/attributions, scenario diffs) passed as **XML‑tagged sections** (Claude supports XML tagging). 

- **Output schema:** JSON with `drivers[]`, `evidence[] {snippet, url}`, `counterfactual[]`, `confidence {p10,p50,p90,consensus,anomaly}`.

## 12) Orchestration, MLOps & Monitoring

- **Pipelines:**  
  - **Data ingestion** (EventBridge → Lambda → S3/Glue).   
  - **Model retraining** schedule (weekly/monthly); **Clarify** processing jobs for updated attributions; store outputs to S3.   
  - **KB sync** jobs ingest new releases/docs into OpenSearch Serverless. 

- **Drift monitoring:**  
  - **Clarify explainability drift** and (optional) **bias metrics** (if any sensitive attributes were modeled, typically not applicable for macro indicators).

## 13) Security, Privacy & Cost

- **IAM least privilege** for Lambdas, Bedrock, OpenSearch Serverless, S3; encryption at rest (S3/AOSS), HTTPS in transit. (OpenSearch Serverless policies for vector collection access.)   
- **No PII**; all data public (FRED/BEA).  
- **Cost controls:**  
  - Batch forecasting and Clarify jobs off‑peak; Bedrock **context windows** sized to needed snippets; OpenSearch Serverless auto‑scales for vector workload.

## 15) Measuring Success (Hackathon Judging)

- **Usability & insights:** Task‑completion time; clarity scores for explanations; click‑through to evidence links.  
- **Clarity of AI explanations:** % of claims with **citations**, user rating of comprehensibility, and **coverage** of the four lenses. (KB returns **citations** automatically when used with Bedrock.)

## 16) Why This Will Stand Out

- **Faithful & Lay‑friendly:** Cards explain *why* the forecast moved, *what evidence supports it*, *what could flip it*, and *how certain we are*.  
- **Evidence‑grounded by design:** Every claim cites BEA/FRED sources retrieved through **Bedrock Knowledge Bases**; the LLM can’t “freewheel.”   
- **Scenario agility:** Analysts can quickly test policy or macro shocks with revision‑aware baselines (ALFRED), making counterfactuals **credible**. 

---

## 17) Implementation Notes & Choices

- **Forecast path selection:** If your AWS account can use **Amazon Forecast**, the AutoPredictor + Explainability is quick and robust; otherwise, the **SageMaker** path gives full control and remains fully AWS‑native. (Forecast availability caveat included.)   
- **Exogenous variables:** Treat release calendars as “known future events” and apply scenario shocks as **forward‑looking related time series**; keep schema compatible for future migration to Forecast if desired.   
- **Explainability granularity:** Use Clarify **timewise** and **featurewise** outputs to populate driver cards per horizon step (e.g., next month vs next quarter).

## 18) Demo Flow (what the judges will experience)

1. Select **Inflation (CPI)** → see historical trend with **next release date**.   
2. Click **Forecast** → p10/p50/p90; “Top drivers” list (e.g., **energy**, **services inflation**) from Clarify.   
3. Open **Explainability Card** →  
   - **Drivers:** “Energy CPI contributed +0.15pp to next‑month inflation p50.”  
   - **Evidence:** Snippets from **BLS/FRED release via FRED** page, **BEA PCE** note, each with a citation link.   
   - **Counterfactual:** “If gasoline prices +5% next month, p90 hits 3.7%.” (Computed from scenario.)  
   - **Confidence:** “Tight bands; model consensus high; no anomalies.” (RCF/quantiles.)   
4. Try **Scenario Studio**: Raise unemployment by 0.3pp → reforecast → updated card with deltas.  
5. Trigger **alerts** by loading newly released data (simulated EventBridge event → SNS email).
